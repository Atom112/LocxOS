# LocxOS 开发计划（从最小内核到文件系统与进程调度）

本文档用于指导后续学习与开发，目标是以“可运行、可验证、可迭代”为原则，逐步把当前最小内核扩展为具备基础操作系统能力的教学内核。
参考资料：https://wiki.osdev.org/Expanded_Main_Page

## 1. 总体原则

1. 每个阶段必须有可观测结果（屏幕输出、串口日志、测试命令、断点命中）。
2. 每次只引入一类复杂性（例如先中断，再调度；先物理内存，再虚拟内存扩展）。
3. 每个阶段结束都要保留可回退标签（建议 Git tag）。
4. 优先保证“调试能力”再做“功能复杂度”。
5. 后续接口设计优先保持 POSIX 兼容方向（命名、语义、错误码与行为约定），避免引入与 POSIX 明显冲突的内核/用户态接口，以便未来 GUI 与应用生态接入。
6. 文档按版本归档：`doc/` 目录将以版本号（如 `v0.0.1`）为单位划分基础知识子目录。随版本迭代，将专门在该版本的目录下补充新增功能的相关深度知识和原理解析，方便初学者由浅入深跟随学习。

## 2. 阶段路线总览

- 当前状态：M0 已完成（可引导、进入 64 位、C 入口、VGA/串口输出）
- M1：中断与异常基础
- M2：时间基准与内核时钟
- M3：物理内存管理（PMM）
- M4：虚拟内存管理（VMM）
- M5：内核堆与基础容器
- M6：进程/线程基础与上下文切换
- M7：抢占式调度
- M8：系统调用与用户态最小框架
- M9：块设备抽象与磁盘 I/O
- M10：文件系统（先只读再读写）
- M11：用户态程序加载与运行
- M12：稳定性与可观测性增强

## 3. 详细迭代计划

## M0（已完成）最小可运行闭环

目标：GRUB + Multiboot2 -> long mode -> kmain -> VGA/串口输出。

验收标准：

1. `make run` 能看到 `LocxOS boot ok`。
2. `make run-serial` 能看到串口日志。
3. `make debug` + `make gdb` 可断到 `kmain`。

## M1 中断与异常基础（建议 1-2 周）

目标：建立 IDT，能捕获并输出常见 CPU 异常。

子任务：

1. 定义 IDT 表结构与加载函数（`lidt`）。
2. 为 0~31 号异常向量提供统一入口（汇编存根 + C 分发）。
3. 输出异常号、错误码、RIP/CS/RFLAGS 到串口。
4. 增加不可恢复异常时的停机策略（panic）。

验收标准：

1. 人为触发除零异常能稳定打印异常信息。
2. 页面错误（#PF）能打印 fault 地址（CR2）。

## M2 时间基准与内核时钟（建议 1 周）

目标：获得周期性中断，为调度与超时机制做准备。

子任务：

1. 初始化 PIT（或 APIC Timer，教学阶段建议先 PIT）。
2. 建立 tick 计数器（全局时钟滴答）。
3. 串口周期输出 tick（先低频输出，避免刷屏）。

验收标准：

1. 每秒 tick 计数符合预期频率。
2. 中断处理后系统稳定，不出现随机死机。

## M3 物理内存管理 PMM（建议 1-2 周）

目标：管理可用物理页帧，支持页级分配与释放。

子任务：

1. 解析引导器提供的内存布局（Multiboot2 memory map）。
2. 建立页帧位图或伙伴系统（先位图更易落地）。
3. 提供 `pmm_alloc_page` / `pmm_free_page`。

验收标准：

1. 启动时打印总内存、可用页、已占用页。
2. 连续分配/释放压力测试无明显错误。

## M4 虚拟内存管理 VMM（建议 2 周）

目标：在已有 PMM 基础上提供页表映射接口。

子任务：

1. 抽象 `map/unmap/translate` 接口。
2. 支持按页映射（4 KiB）并保留大页能力。
3. 引入内核高半区映射策略（可作为进阶目标）。

验收标准：

1. 可动态映射一页并读写验证。
2. 非法访问触发 #PF 且能定位地址。

## M5 内核堆与基础容器（建议 1 周）

目标：从“页分配”提升到“小块动态分配”。

子任务：

1. 实现早期 bump allocator（启动期）。
2. 实现通用 `kmalloc/kfree`（先 slab-like 或分离链表实现）。
3. 补齐基础数据结构（链表、环形队列、位图工具）。

验收标准：

1. `kmalloc/kfree` 小规模压力测试通过。
2. 内核关键模块不再依赖静态数组硬编码。

## M6 进程/线程基础与上下文切换（建议 2 周）

目标：实现最小执行实体（建议先内核线程）。

子任务：

1. 定义任务控制块 TCB/PCB。
2. 实现上下文保存与恢复（寄存器、栈）。
3. 创建两个测试线程轮流执行。

验收标准：

1. 两个内核线程可重复切换并输出不同日志。
2. 栈切换稳定，无异常覆盖。

## M7 抢占式调度（建议 2 周）

目标：基于时钟中断进行任务抢占。

子任务：

1. 调度策略先用 Round-Robin。
2. 在 timer IRQ 中触发调度点。
3. 维护就绪队列、当前任务、空闲任务。

验收标准：

1. CPU 密集任务不会独占系统。
2. 多任务日志交替输出，节奏与 tick 对齐。

## M8 系统调用与用户态最小框架（建议 2 周）

目标：打通用户态到内核态的受控入口。

子任务：

1. 建立 syscall ABI（建议 `syscall/sysret` 或 `int 0x80` 先易后优）。
2. 实现基础系统调用：`write`, `exit`, `yield`（接口语义与返回约定尽量对齐 POSIX）。
3. 构建最小用户程序镜像并运行。

验收标准：

1. 用户程序可调用 `write` 输出到串口/屏幕。
2. 用户程序崩溃不会直接拖垮内核（至少可 panic 定位）。

## M9 块设备抽象与磁盘 I/O（建议 2 周）

目标：为文件系统准备统一块设备接口。

子任务：

1. 定义 block device 抽象（read/write sector）。
2. 先接入 RAM disk（简化验证），再考虑 ATA/virtio-blk。
3. 增加缓存层雏形（可选）。

验收标准：

1. 可读写固定扇区并校验数据一致性。
2. 错误路径有日志输出。

## M10 文件系统（建议 3-4 周）

目标：从只读到读写，逐步实现 VFS 雏形。

建议策略：

1. 先实现简单只读文件系统（例如 tarfs 或简化 ramfs）。
2. 再实现可写文件系统（建议先 ramfs，再考虑 FAT32）。

子任务：

1. 文件对象与 inode/dentry 基础结构。
2. 路径解析（`/a/b/c`）。
3. 文件 API：`open/read/write/close`（错误码与基础行为遵循 POSIX 约定）。

验收标准：

1. 能列目录、读取文件内容。
2. 可写阶段能创建文件并再次读取验证。

## M11 用户态程序加载与运行（建议 2-3 周）

目标：支持从文件系统加载用户程序（ELF）。

子任务：

1. ELF 解析与段装载。
2. 用户栈构建（参数传递可后置）。
3. 与调度器结合运行多个用户任务。

验收标准：

1. 至少 2 个用户程序可运行并输出。
2. 程序退出后资源可回收。

## M12 稳定性、工具链与可观测性（持续）

目标：把“能跑”提升为“能查、能测、能持续迭代”。

子任务：

1. 统一日志等级（INFO/WARN/ERROR/PANIC）。
2. 增加简单自测命令或内核测试入口。
3. 建立文档更新规范（每个模块配一页设计说明）。
4. 优化 Makefile：分离 debug/release 配置。

验收标准：

1. 新功能上线有最小回归测试。
2. panic 信息可定位到具体模块和阶段。

- v0.2：M1 + M2

## 6. 推荐当前下一步

建议立刻进入 M1（中断与异常基础），因为这是后续内存管理、调度、系统调用的共同依赖。

最小落地顺序：

1. 先做 IDT 加载 + 默认异常处理。
2. 再做 #DE、#GP、#PF 三个重点异常的详细打印。
3. 最后接入 panic 统一处理路径。
