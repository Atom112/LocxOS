# Linker 脚本相关知识

C 代码和汇编代码经过编译器处理后，会变成单独的一个个目标文件（.o 文件）。怎样把它们拼装成一个**可以被 GRUB 正确加载和运行**的完整操作内核呢？这正是链接器（Linker, 我们用的是 GNU ld）的工作。

链接器行为是由“链接器脚本”（Linker Script）来控制的。在本项目里，这个文件是 kernel/arch/x86_64/linker.ld。

本篇向初学者解释本项目所用脚本里的主要逻辑。

## 1. 输出格式与入口点

`ld
OUTPUT_FORMAT(elf64-x86-64)
ENTRY(_start)
`

1.  **OUTPUT_FORMAT**：指定链接器最终产生的文件应当是 elf64-x86-64 格式。这是当前 Linux 和很多 Unix 环境采用的标准二进制文件执行格式（Executable and Linkable Format, ELF），它能完美封装并区分我们的不同数据段，并且 GRUB 从 0.9.x 时代就开始原生认识 ELF 格式的文件。
2.  **ENTRY**：这里设置整个内核开始执行的起点函数是谁。_start 就是我们在 boot.s 中定义的全局符号。

## 2. 内存布局（SECTIONS）

SECTIONS 块是链接器脚本中最关键的部分。它指示了链接器应当把不同的代码和数据放到加载地址内存的何处。

### 当前地址计数器 "." (Dot)

`ld
. = 1M;
`

链接器中，点号 . 其实就是一个地址指针。. = 1M; 代表“从现在开始，后面分配的段请从物理地址 1MB （也就是 0x100000）处开始放置”。
把内核放在 1MB 偏移处是极其传统的。之所以不放在 1MB 以内（也就是 0x00000 到 0xFFFFF），因为这段被称为“下位内存”（Low Memory），包含硬件被固化的 VGA 显示缓冲区、BIOS 数据区、实模式中断向量表等。如果覆盖掉可能造成极度不可控的主板冲突问题。所以在 1MB 或者更高的安全区域加载内核是标准的做法。

### 各数据段（Sections）摆放

1. **.multiboot 头**：
`ld
.multiboot : ALIGN(8)
{
KEEP(*(.multiboot))
}
`
   之前我们说到 GRUB 会去扫描内核的前 8KB 获取多重启动（Multiboot2）协议头，因此 Multiboot 头必须被最先放置！KEEP() 函数强制链接器如果怀疑这些信息“没用被调用”时千万**不要擅自将它优化掉**。由于对齐要求，被设计按 8 字节（64 位系统下常见）起始对齐。

2. **代码与数据的主要分段 (.text / .rodata / .data / .bss)**：
   汇编和 C 文件里声明的不同类型的内存会被聚集在一起：
   *   .text：放置可执行的 CPU 机器指令（代码段）。
   *   .rodata：放置只读数据（Read Only Data），比如程序里用到的那些常量字符串 "Hello, World!"。
   *   .data：已经赋了初值（比如 int x = 5;）的全局和静态变量。
   *   .bss：未赋初值（默认就是 0，比如 int arr[1024];）的全局变量。这些空间不会出现在编译出的二进制文件实体中，仅指明加载时应该占据的内存大小。这可以极大缩减系统的文件体积。包括像我们在汇编里定义的栈和页表都在这里。

   重点是他们全跟有 ALIGN(4K)。因为内存分页体系里默认的页面大小就是 4096（4KB），如果想要之后分别给各个段做页级权限控制保护（比如禁止向 .text 段写入数据），将所有的区域起始点都卡在 4KB 对齐的页边界上是最好的设计。

### 丢弃无用的段 (/DISCARD/)

`ld
/DISCARD/ :
{
*(.comment)
*(.eh_frame*)
}
`
普通的 C 编译器会在结果文件里混进包含编译器版本之类的注释信息段（.comment）或者异常抛出栈展开的相关帧处理数据（.eh_frame）。由于我们的系统仅仅是个裸机微核，根本用不到甚至也不理解这些系统级高层特性。因此可以直接命令直接扔掉它们，保持内核的绝对精简。
